## 목적

이 문서는 현재의 `2재료 조합 시트`를 `배우고 축적하는 조합 도감`으로 확장하고, 플레이어가 직관적으로 중요하다고 느끼는 `라이터`를 실제 핵심 도구로 승격하기 위한 설계다.

이번 설계의 핵심 목표는 세 가지다.

- 조합이 너무 막막하게 느껴지지 않게, 배운 내용을 축적하는 `도감`을 제공한다.
- 책/메모를 단순 잡템이 아니라 `지식 해금 아이템`으로 만든다.
- 라이터를 설명만 그럴듯한 잡템이 아니라, 실제 조합 가치가 있는 `잔량 기반 점화 도구`로 만든다.

즉 이번 설계는 `레시피 수를 더 늘리는 작업`이 아니다.

- 이미 존재하는 레시피 전체를 도감의 마스터 목록으로 본다.
- 플레이어가 그중 일부를 배우고, 일부는 실험으로 밝혀낸다.
- 특정 조합은 `재료 2개만 있으면 되는 조합`이 아니라 `도구까지 갖춰야 되는 조합`으로 재정의한다.

## 현재 진단

현재 구현에는 공용 조합 시트와 데이터 기반 조합 결과가 이미 있다.

- 실내와 실외에서 같은 조합 시트를 열 수 있다.
- `crafting_combinations.json`이 결과물과 시간 비용을 정의한다.
- `RunState.attempt_craft(...)`가 실제 인벤토리 변경을 처리한다.

하지만 현재 플레이 감각에는 두 가지 큰 공백이 있다.

### 1. 조합 안내 계층이 거의 없다

- 조합 시트는 `재료 두 개 선택 -> 결과 문구`에 가깝다.
- 배운 레시피를 다시 확인할 구조가 없다.
- 플레이어는 “세상에 어떤 조합이 존재하는지”를 거의 감으로만 추측해야 한다.

그 결과 조합은 흥미롭기보다 답답하게 느껴질 수 있다.

### 2. 라이터가 기대만큼 가치가 없다

- 라이터는 아이템 정의만 있고, 실제 조합 시스템에서 거의 참조되지 않는다.
- 현재 구조에서는 라이터가 `설명상 쓸모 있어 보이는 아이템`이지 `실제로 필요한 아이템`이 아니다.

플레이어는 라이터를 보며 즉시 “이건 중요하다”고 느끼는데, 시스템은 그 기대를 지지하지 못한다.

## 목표 경험

플레이어는 다음 경험을 받아야 한다.

1. 조합 시트를 열면 `직접 조합`과 `조합 도감`이 분리되어 있어 정보가 덜 막막하다.
2. 도감에는 처음부터 전체 조합 슬롯이 존재하고, 아직 모르는 조합은 `???`로 보인다.
3. 어떤 조합을 우연히 성공시키면, “내가 알아냈다”는 감각으로 해당 도감 슬롯이 열린다.
4. 책/메모를 읽으면 그 문서가 담당하는 조합 묶음이 한 번에 열린다.
5. 라이터를 얻었을 때 즉시 “이제 실제로 불을 쓸 수 있다”는 전환점이 생긴다.
6. 라이터는 무한 도구가 아니라 `잔량`이 있어 언제 점화를 시도할지 판단하게 만든다.

## 범위

이번 설계는 아래를 포함한다.

- 공용 조합 시트의 `직접 조합 / 조합 도감` 탭 분리
- 전체 레시피 마스터 목록을 기반으로 한 조합 도감
- `조합 성공 시 자동 해금`
- `책/메모 읽기 시 고정된 레시피 묶음 해금`
- 라이터의 `잔량` 시스템
- 특정 조합의 `required_tool_ids` / `tool_charge_costs`
- 라이터 잔량과 도구 조건의 UI 노출

이번 설계에서 의도적으로 제외하는 것:

- 다른 모든 도구의 내구도/잔량 전면 도입
- 영구 세이브/메타 진행과의 완전 통합
- 별도 독립 도감 화면
- 실패 조합 전부의 기록 도감화

즉 이번 라운드는 `도감의 첫 완성형`과 `라이터의 실제 가치 부여`에 집중한다.

## 선택된 접근

이번 설계는 다음 접근을 선택한다.

- 공용 조합 시트 안에서 `직접 조합` / `조합 도감` 탭 분리

이 접근을 선택한 이유:

- 별도 도감 화면으로 가면 1차 범위가 커진다.
- 기존 공용 조합 시트 위에 얹으면 실내/실외 공용 구조를 그대로 유지할 수 있다.
- 현재 플레이어의 답답함은 “조합을 어디서 보지?”보다 “조합 시도와 확인이 같은 흐름 안에 있어야 한다”에 더 가깝다.

## 도감 구조

### 도감은 전체 레시피 마스터 목록이다

도감에는 처음부터 전체 조합 슬롯이 존재한다.

- 구현상 마스터 목록은 `crafting_combinations.json`에서 직접 읽는다.
- 별도의 수동 도감 목록 파일은 1차에서 만들지 않는다.

즉 도감은 `새 데이터를 따로 만드는 시스템`이 아니라 `기존 레시피 데이터를 플레이어 지식 상태와 결합해 렌더링하는 시스템`이다.

### 카테고리별로 묶는다

도감은 아래 카테고리로 묶는다.

- `불 / 열`
- `음식 / 음료`
- `위생 / 의료`
- `수리 / 보강`

카테고리는 레시피 데이터의 `codex_category`로 결정한다.

### 미발견 레시피는 완전히 숨긴다

미발견 레시피는 도감에서 단순히 `???`로만 보인다.

- 재료를 보여주지 않는다.
- 결과물을 보여주지 않는다.
- 설명도 보여주지 않는다.
- 조건도 보여주지 않는다.

즉 플레이어는 “이 카테고리에 아직 모르는 무언가가 있다”만 안다.

### 발견된 레시피는 실용 정보만 보여준다

발견된 레시피는 다음 정보를 보여준다.

- 재료 2개
- 결과물
- 짧은 설명
- 조건
  - 예: `실내 전용`
  - 예: `도구: 라이터`

도감은 정답집이 아니라, 이미 배운 조합을 다시 참고하는 실용 노트여야 한다.

## 해금 규칙

### 1. 조합 성공 시 자동 해금

플레이어가 어떤 조합을 성공시키면 해당 레시피는 즉시 도감에 기록된다.

- 이미 알고 있던 레시피라면 변화 없음
- 모르던 레시피라면 그 슬롯이 열림

즉 `우연한 성공`도 지식의 축적이다.

### 2. 책/메모 읽기 시 고정 묶음 해금

책/메모는 `고정된 recipe id 묶음`을 해금한다.

- 각 문서는 `knowledge_recipe_ids`를 가진다.
- 랜덤 카테고리 해금은 쓰지 않는다.
- 같은 책이 무엇을 가르치는지 정체성이 분명해야 한다.

예:

- `즉석 보온 메모`: 초기 불/열 레시피 묶음
- `간이 취사 메모`: 온수, 차, 수프 계열
- `현장 위생 메모`: 세척, 위생 키트 계열

### 3. 책/메모는 비소모 지식 아이템이다

책/메모는 읽어도 사라지지 않는다.

- 한 번 읽으면 지식만 해금된다.
- 아이템 자체는 인벤토리에 남는다.
- 같은 책을 다시 읽어도 추가 해금은 없다.

## 라이터 잔량 시스템

### 1차 대상은 라이터만

이번 라운드는 `잔량 시스템` 자체는 일반화하되, 실제 사용 대상은 `라이터` 하나만 둔다.

아이템 인스턴스가 가질 필드:

- `charges_current`
- `charges_max`

아이템 정의가 가질 기본 필드:

- `charges_max`
- `charge_label`

표시는 `잔량 5/5` 형태로 간다.

### 라이터는 재료가 아니라 도구다

라이터는 조합 재료로 소모되지 않는다.

대신 특정 레시피의 `required_tool_ids`로만 사용된다.

그래서 설계상 흐름은 이렇게 나뉜다.

- `조립`: 재료 2개만으로 가능
  - 예: `신문지 + 식용유 -> 고농축 땔감`
  - 예: `철제 식품 캔 + 고농축 땔감 -> 깡통 화로`
- `점화 / 가열`: 라이터가 필요
  - 예: `생수 + 깡통 화로 -> 뜨거운 물`
  - 예: `생수 + 양초 랜턴 -> 뜨거운 물`
  - 예: `생수 + 차폐 양초 -> 뜨거운 물`

이렇게 해야 라이터가 실제 가치 있는 전환점 아이템이 된다.

### 라이터 잔량 소모

도구가 필요한 조합은 `tool_charge_costs`를 가진다.

예:

- `{"lighter": 1}`

성공 시:

- 레시피는 해금된다.
- 라이터 잔량이 감소한다.

실패 시:

- 라이터가 없으면 `라이터가 필요하다`
- 라이터 잔량이 부족하면 `라이터 잔량이 부족하다`

1차에서는 도구 조건 미충족 상태에서 레시피 자체는 무효 처리하되, 이미 알고 있는 레시피의 조건은 그대로 보여준다.

## 데이터 구조

### 레시피 데이터

`crafting_combinations.json`에 아래 필드를 추가한다.

- `codex_category`
- `required_tool_ids`
- `tool_charge_costs`
- 필요 시 `codex_order`

기존 `ingredients`, `contexts`, `ingredient_rules`, `result_items` 구조는 유지한다.

### 책/메모 아이템 데이터

`items.json`에 책/메모 계열을 추가한다.

필수 필드:

- `readable: true`
- `knowledge_title`
- `knowledge_recipe_ids`

선택 필드:

- `usage_hint`
- `cold_hint`

### 런타임 상태

`RunState`에 아래 상태를 둔다.

- `known_recipe_ids`
- `read_knowledge_item_ids`

1차에서는 `런 단위 상태`로 두고, 이후 저장 시스템이 붙을 때 영구화한다.

즉 이번 설계는 메타 세이브까지 한 번에 하지 않는다.

## UI 구조

### 공용 조합 시트

공용 조합 시트는 두 탭으로 나뉜다.

- `직접 조합`
- `조합 도감`

#### 직접 조합 탭

- 지금처럼 재료 2개를 선택해 시도
- 결과 카드에 조건 미충족 문구를 명확히 보여줌
- 알려진 레시피라면 결과 카드에서 조건을 더 분명히 보여줄 수 있음

#### 조합 도감 탭

- 카테고리별 리스트
- 각 카테고리에 `발견 수 / 전체 수`
- 미발견은 `???`
- 발견된 항목은
  - 재료
  - 결과물
  - 짧은 설명
  - 조건

### 인벤토리 / 아이템 상세

- 라이터는 상세에 `잔량 5/5`를 표시
- 라이터 태그는 `점화 도구`를 드러낸다
- 책/메모는 `읽는다` 액션이 생긴다
- 이미 읽은 책은 다시 읽어도 “이미 아는 내용이다” 수준의 피드백만 준다

## 1차 구현 범위

이번 라운드의 첫 구현은 아래만 포함한다.

- 조합 도감 탭
- `???` 미발견 슬롯
- 조합 성공 시 자동 해금
- 책/메모 아이템 3종 내외
- 책/메모 읽기 액션
- 라이터 `잔량`
- 라이터가 필요한 가열 계열 레시피 몇 개

즉 1차는 `도감 시스템의 첫 완성형`과 `라이터의 체감 가치 부여`가 목표다.

## 검증 기준

다음이 검증돼야 한다.

- 전체 레시피가 카테고리별 도감 목록으로 안정적으로 렌더링된다.
- 미발견 레시피는 `???`만 보인다.
- 조합 성공 시 해당 레시피가 해금된다.
- 책/메모를 읽으면 해당 묶음이 해금된다.
- 같은 책을 다시 읽어도 중복 해금되지 않는다.
- 라이터 잔량이 보인다.
- 라이터가 필요한 레시피는 라이터 없으면 실패한다.
- 라이터 사용 후 잔량이 감소한다.

## 후속 확장 여지

이 설계는 이후 아래로 확장 가능해야 한다.

- 성냥, 랜턴, 배터리 장비 같은 다른 도구 잔량/내구도
- 실패 조합 기록의 도감화
- 아이템별 실험 노트형 세부 도감
- 런타임 지식의 영구 저장
- 책/메모 rarity와 업종별 지식 분포

하지만 이번 라운드는 그 확장 기반만 정확히 깔고 끝낸다.
