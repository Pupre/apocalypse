## 목적

이 문서는 `SurvivalSheet` 기반 실내 세로 전환 설계 위에, 실제 조작감이 더 자연스럽도록 `조합 UI`를 다시 다듬기 위한 보완 설계다.

핵심 목표는 세 가지다.

- `가방`이 메인이고 `조합`은 그 안에서 잠깐 쓰는 기능이라는 위계를 지킨다.
- `easy` 모드의 도움 기능을 유지하되, 모든 조합 시도는 계속 허용한다.
- 조합 성공 직후 결과물을 같은 아이템 상세 흐름 안에서 보게 하여 창 왕복을 없앤다.

## 확정 원칙

### 1. 가방 우선

- 플레이어가 보고 있는 기본 맥락은 언제나 `가방 목록 + 선택 아이템 상세`다.
- 조합은 별도 도구창이나 별도 시트의 주인공이 아니다.
- 조합 상태는 얇은 상태줄과 보조 행동으로만 드러난다.

### 2. 모든 조합 시도 허용

- 조합은 `맞는 레시피만 고를 수 있는 퍼즐 UI`가 아니다.
- 플레이어는 비호환 재료도 그대로 선택하고 시도할 수 있다.
- 이 원칙은 `허탕`, `시간 낭비`, `위험한 실험` 같은 게임 감각을 유지하기 위해 필요하다.

### 3. Easy는 안내만 하고 차단하지 않는다

- `easy` 모드에서는 호환 재료를 하이라이트한다.
- 하지만 하이라이트되지 않은 재료도 여전히 탭 가능하고 조합 시도 가능하다.
- `hard` 모드에서는 이 하이라이트만 끈다. 조합 흐름 자체는 동일하다.

### 4. 카피는 항상 중립적이어야 한다

- 시도 전 버튼은 항상 `조합 시작`, `조합`, `취소`처럼 중립적으로 유지한다.
- `허탕 조합`, `성공 가능`, `실패 예상` 같은 선제적 판단 카피는 금지한다.
- 성공/실패 정보는 시도 이후 피드백으로만 드러난다.

## 현재 문제 진단

이전 실험안들은 아래 문제를 드러냈다.

- 상시 `조합 모드` 토글은 가방보다 조합 도구를 먼저 보게 만든다.
- `슬롯 A / 슬롯 B` 문법은 기능은 명확하지만 모바일에서 너무 기계적으로 느껴진다.
- 결과 미리보기는 플레이어의 실험 감각을 약하게 만든다.
- 별도 조합창 중심 흐름은 결과물을 이해하기 위해 다시 가방으로 돌아가야 해서 조작감이 끊긴다.

이번 보완 설계는 위 네 문제를 동시에 피한다.

## 선택된 상호작용 문법

### 기본 상태

가방 탭의 평상시 구조는 그대로 유지한다.

- 상단: 현재 탭 제목과 가벼운 상태 문구
- 중단: 아이템 목록
- 하단: 선택 아이템 상세

상세의 기본 주인공은 기존 행동이다.

- `사용`
- `장착`
- `읽는다`
- `버린다`

`조합 시작`은 이들과 같은 상세 영역 안에 존재하지만, 시각적으로는 한 단계 낮은 보조 행동이다.

즉 플레이어가 아이템을 열었을 때 가장 먼저 느껴야 하는 것은 `이 아이템을 이해하는 것`이지 `지금부터 조합 퍼즐을 푸는 것`이 아니다.

### 조합 시작

플레이어가 선택 아이템 상세에서 `조합 시작`을 누르면, 그 아이템이 이번 시도의 첫 재료가 된다.

조합 시작 시 변화는 최소화한다.

- 가방 목록은 그대로 유지한다.
- 선택 아이템 상세도 같은 바텀시트 안에 남아 있다.
- 상단에는 얇은 상태줄만 추가된다.
- 상태줄은 `조합 중: 신문지` 같은 현재 기준 재료만 보여준다.

즉 플레이어는 `새 창으로 들어갔다`고 느끼지 않고, 같은 가방 안에서 맥락만 잠깐 바뀐다고 느껴야 한다.

### 조합 중 아이템 선택

조합 시작 이후에도 플레이어는 가방의 아무 아이템이나 그대로 탭할 수 있다.

- `easy`에서는 호환 재료만 더 눈에 띄게 하이라이트한다.
- 비호환 재료도 탭 가능하다.
- 탭하면 그 아이템의 상세가 같은 바텀시트에 열린다.

이때 중요한 점은 `선택 가능 여부`가 아니라 `안내 강도`만 달라져야 한다는 점이다.

### 조합 확정

두 번째 아이템을 탭해도 즉시 조합되지는 않는다.

- 결과 미리보기는 없다.
- 상세 시트의 주 행동 버튼만 `조합`으로 바뀐다.
- 플레이어는 현재 보고 있는 두 번째 재료의 설명을 읽은 뒤 스스로 시도를 확정한다.

이 문법은 세 가지를 동시에 만족시킨다.

- `즉시 조합`보다 실수 비용이 낮다.
- `결과 미리보기`보다 실험 감각이 강하다.
- 여전히 가방 상세 문법 안에 머문다.

## 결과 처리

### 성공

조합 성공 시:

- 현재 조합 상태는 종료된다.
- 결과 아이템이 같은 상세 시트에 즉시 열린다.
- 플레이어는 별도 창 전환 없이 방금 만든 아이템의 설명과 행동을 바로 읽을 수 있다.

즉 성공의 리워드는 아이템 생성뿐 아니라 `끊기지 않는 이해 흐름`이어야 한다.

### 실패

조합 실패 시:

- 현재 보고 있던 상세 시트는 그대로 유지된다.
- 상단 상태줄과 결과 피드백 영역에 실패 메시지만 갱신된다.
- 조합 상태는 유지한다.

즉 실패는 `조합 모드가 풀려버리는 번거로운 벌칙`이 아니라, `같은 맥락 안에서 다음 실험을 이어갈 수 있는 피드백`이어야 한다.

## Easy / Hard 계약

### Easy

- 호환 재료를 하이라이트한다.
- 하이라이트는 현재보다 확실하게 보이게 한다.
- 단, 이것은 추천/안내일 뿐이며 입력 제한을 만들지 않는다.

### Hard

- 호환 재료 하이라이트를 끈다.
- 나머지 조합 흐름은 Easy와 완전히 동일하다.
- 즉 `조합 가능/불가능을 몸으로 기억해야 하는 차이`만 발생한다.

## 시각적 우선순위

조합 상태가 보이더라도 화면의 시각적 우선순위는 아래 순서를 유지한다.

1. 가방 목록
2. 선택 아이템 상세
3. 현재 가능한 기본 행동
4. 조합 상태줄

즉 `조합 중`이라는 사실이 가방 자체보다 더 큰 존재감을 가져서는 안 된다.

## 금지 사항

이번 보완 설계에서는 아래를 금지한다.

- 상시 화면 상단의 거대한 조합 토글
- `슬롯 A / 슬롯 B` 같은 도구창형 명명
- 결과 미리보기
- 성공/실패를 시도 전에 암시하는 버튼/문구
- 호환 재료 외 선택 자체를 막는 입력 제한
- 조합 성공 후 별도 조합창에 머물러 결과를 다시 찾게 만드는 흐름

## 구현 메모

구현 시 실제 상태는 아래 정도면 충분하다.

- `selected_item_id`
- `craft_anchor_item_id`
- `craft_candidate_item_id`
- `is_crafting_context_active`

그리고 UI 계약은 아래로 정리할 수 있다.

- 기본 상세에서 `조합 시작`
- 조합 중 상세에서 `조합` 또는 `취소`
- 성공 시 `selected_item_id = crafted_result_id`
- 실패 시 `selected_item_id` 유지

## 완료 기준

아래가 모두 만족되면 이번 보완 설계는 구현 완료로 본다.

- 플레이어가 가방 상세에서 `조합 시작`으로 진입할 수 있다.
- 목록은 그대로 유지되며, easy 모드에서는 호환 재료만 확실히 하이라이트된다.
- 하이라이트되지 않은 재료도 선택하고 조합 시도할 수 있다.
- 결과 미리보기 없이 `조합` 버튼으로만 시도를 확정한다.
- 성공 시 결과물 상세가 같은 시트에 즉시 열린다.
- 실패 시 현재 상세는 유지되고 피드백만 갱신된다.
- 플레이 감각이 `조합 도구 사용`이 아니라 `가방 안에서 실험`처럼 느껴진다.
